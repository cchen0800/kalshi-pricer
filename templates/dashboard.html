How to use this dashboard

A logger that prices every Kalshi BTC hourly strike under a zero-drift lognormal and shows you where the model and the market disagree. It does not place trades — you do that by hand on Kalshi if you decide an edge is real.

What you're looking at

Every 30 seconds the engine fetches the nearest open hourly event from Kalshi (e.g. KXBTCD-26APR2723 = April 27, 2026 at 11pm EDT), pulls BTC spot + 60 1-min candles from Coinbase, computes annualized realized vol, prices each strike, and writes it all to pricer.db. The table you see is the latest poll, ascending by strike. The ATM band (strikes within $250 of spot) is bolded — that's where the action lives.

The header strip is your ground truth: the live tag is green when fresh, yellow when poll older than 90 seconds (engine may be stuck). When the hour closes, the engine automatically rolls forward to the next hourly — you'll just see the event ticker change.

How the stats work

The model uses a deliberately small set of inputs: spot, realized vol, and time to close. Every poll it takes Coinbase spot, looks at the last 60 one-minute closes, converts those into 1-minute log returns, measures their standard deviation, annualizes that into realized σ, and then prices each strike from those inputs.

In other words: this dashboard is not trying to predict where BTC wants to go. It is trying to estimate how much BTC could reasonably move before settlement if the next few minutes behave like the recent past.

  • Spot: the current Coinbase BTC price.
  • Realized σ: the volatility implied by the last 60 one-minute closes, annualized for convenience.
  • Minutes left: how much time remains until the Kalshi hourly market closes.
  • Model %: the probability that BTC finishes above a given strike, using those three inputs.

What "zero-drift lognormal" means

Lognormal means we model future BTC moves in percentage terms rather than absolute dollars, and we keep price strictly positive. That is a standard short-horizon asset model because a $500 move means something different at $20k BTC than at $100k BTC.

Zero drift means the model does not assume BTC is biased upward or downward over the next few minutes. It is intentionally neutral about direction. The model says, effectively, "I have a view on how noisy the path may be, but not on whether the next move is up or down."

One subtle point: zero drift does not mean "price is frozen" or "there is no movement." It only means there is no directional forecast baked in. Volatility still spreads out the possible outcomes, and that spread is what drives the strike probabilities.

The model formula includes a small negative-looking adjustment term inside the exponent. That is not a bearish view. It is the standard correction that keeps the expected future price centered on spot even when volatility is present. So in practical terms, zero drift here means: no trend call, only a volatility call.

We use this on purpose because it gives us a clean baseline. It is fast, easy to reason about, hard to overfit, and useful for spotting where the market is disagreeing with a simple "recent vol + no directional bet" distribution. When the market disagrees because BTC is genuinely trending, this model will be wrong — and that is exactly one of the main failure modes to watch for.

Filter buttons

Above the grid, click to slice the view:

  • All — every strike in the event (~190 typically).
  • ITM (strike < spot) — YES is currently winning. Top of the ladder.
  • ATM ±$500 — strikes near spot. The action zone.
  • OTM (strike ≥ spot) — YES is currently losing. Bottom of the ladder.
  • Flagged — rows where the lift edge exceeds threshold. If this count is 0, there's nothing actionable right now.

Counts on each button update with every poll, so you can see "5 flagged" without filtering.

The intended workflow

  1. Open this page next to the Kalshi web app. Both show the same strikes — Kalshi has Strike + Chance + Yes/No, this dashboard adds Model + Edge + Action.
  2. Wait for the sweet spot: T-15 to T-3 minutes before close.
    • Earlier: prices haven't converged on the right strike, lognormal is noisy.
    • Later (T < 3 min): Kalshi settles on a 60-second BRTI average, our pricer assumes a single terminal point — the model is wrong here, not the market.
  3. Click "Flagged". If the count is 0, no edge worth chasing. Move on.
  4. For a flagged row, sanity check before acting:
    • Is Size > 100? Below that you might lift one order with nothing behind it.
    • Is the strike near spot? Tail strikes give bogus edges (model says 0.001%, market floor is 1¢ → "+0.5¢ edge" that isn't real).
    • Does the edge survive 2-3 polls? Auto-refresh is 2s. Persistent edges are more believable than bid/ask jitter.
    • Does the direction match a story? "Market thinks tails are fatter than lognormal" makes sense. "Market thinks ATM is biased upward" might just mean BTC is trending and our zero-drift model is off.
  5. If you decide to act: copy the Ticker from the rightmost column → open Kalshi → paste into search → place the order by hand in the Kalshi UI. This dashboard never trades for you.

Pitfalls already burned into this design

  • First 5 minutes after launch: realized σ uses the trailing 60 minutes; the most recent minute is partial. Don't trust early σ prints.
  • The 60-second BRTI average: at T < 3 min, the right model is "average of 60 correlated 1-second samples", not "terminal price". Edges this close are usually model-wrong.
  • Coinbase ≠ BRTI: Kalshi settles on CF Benchmarks BRTI (basket across 6 exchanges). We use Coinbase as a proxy. Under stress they can drift several dollars. Treat sub-$10 BTC-denominated edges as basis noise.
  • Tail strikes: deep ITM (model 99.9%) and deep OTM (model 0.1%) often show ±0.5¢ "edges" that are quote-floor artifacts. Look near spot.
  • Bid/ask quote floor: Kalshi can't quote below 1¢, so 0.99/1.00 means "essentially certain to be YES" with 1¢ being mostly tick noise.

Common scenarios

Scenario: Flagged count = 0 for a long time.
The market and model agree. Either the model is well-calibrated for this regime, or the threshold is too tight. Both are fine. Wait or lower the threshold in config.yaml.
Scenario: Symmetric bias near spot (e.g. all ITM strikes flag Buy NO, all OTM flag Buy YES).
The market is more confident than the model — usually because BTC is trending in one direction and our zero-drift assumption is off. Be skeptical of these edges.
Scenario: One strike flags, neighbors don't.
That's a localized bid/ask anomaly — probably worth investigating. A single resting order or a pulled quote can cause it. Check Size before lifting.

What's intentionally not in scope

  • No order placement (read-only Kalshi client).
  • No CF Benchmarks subscription (Coinbase proxy only).
  • No drift / skew / IV blending — pure zero-drift lognormal.
  • No alerts (Slack, push, email). The dashboard is the alert.

Stopping & restarting

Hit Ctrl-C in the terminal where you ran python dashboard.py. Both the HTTP server and the engine thread shut down cleanly. pricer.db persists — when you restart, the engine resumes appending to the same DB.

For headless logging without the UI: python main.py. Same engine, no port, just stdout.
